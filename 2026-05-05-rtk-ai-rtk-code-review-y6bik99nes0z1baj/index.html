結論

本番採用は非推奨。 設計の方向性は妥当だが、誇張・矛盾・実装の詰めの甘さが多数あり、実験的に試す程度に留めるのが安全。

ツール概要

LLM コーディングエージェント（Claude Code / Cursor / Copilot 等 9 種）の PreToolUse に hook を挿し、git status などを rtk git status に書き換えて RTK バイナリを噛ませる。内部は (1) コマンド毎の手書き Rust フィルタ (src/cmds/**)、(2) TOML 宣言の 8 段パイプライン (src/core/toml_filter.rs:16-24：ANSI除去 / regex 置換 / 行フィルタ / truncate 等) を適用して LLM に渡す出力を圧縮。要は「コマンド毎のサマライザ／truncator」で、新規アルゴリズムは無し。

メリット

  • 書き換え判定の真実源が一箇所: src/discover/registry.rs 単独で完結、9 hook はただの delegate。
  • Compound command の allow 判定が厳格: src/hooks/permissions.rs:31-90 で &&/; 連結の全セグメントが allow にマッチしないと Allow にしない。default は exit 3 = ask。
  • Telemetry はデフォルト OFF / 完全 opt-in: src/core/config.rs:91-98、rtk telemetry forget でローカル DB 削除＋サーバ erasure (src/core/telemetry_cmd.rs:109-159)。
  • .semgrep.yml で危険パターン (Command::new($var), unsafe, LD_PRELOAD 等) を ERROR 指定。意識自体は高い。
  • フィルタが TOML 化されており project-local .rtk/filters.toml でカスタマイズ可能。

デメリット／懸念点

高（採用判断に効く）

  • 「zero dependencies」が虚偽: 公式説明と矛盾し Cargo.lock の transitive crates は 203 個、直接依存だけで 22 個 (Cargo.toml:14-38)。「Single Rust binary」までは正しいが zero deps は誤誘導。
  • ライセンス不整合: LICENSE は Apache-2.0、Cargo.toml:7 / Formula/rtk.rb:10 / openclaw/openclaw.plugin.json:7 / README badge は MIT。SPDX が分裂。企業利用 fatal。
  • install.sh にチェックサム検証ゼロ (install.sh:97-107): SHA256/署名なしで ~/.local/bin へ配置。GitHub Release 改竄や中間者攻撃でバイナリ差し替え可能。Formula/rtk.rb:15-32 も PLACEHOLDER_SHA256_INTEL のまま version "0.1.0" で Homebrew tap は実体のない宣伝（README は brew install を推奨）。
  • 自分で禁じた unsafe を自分で違反: .semgrep.yml:13-17 で禁止しているのに src/main.rs:2218-2233 で #[allow(unsafe_code)] を付けて signal handler 内で libc::waitpid を呼ぶ。waitpid は async-signal-safe ではなく UB。
  • ureq の自己ルール違反: .semgrep.yml:67-75 で telemetry 外利用禁止だが src/core/telemetry_cmd.rs:173 で違反。CI が通っているのは lint を緩めている証拠。

中

  • 「60-90% トークン削減」の根拠が脆弱: 削減量計算は text.len() / 4 の固定係数 (src/core/tracking.rs:1283-1286)。実際の Claude/GPT トークナイザを通していないので tiktoken と数十%単位でズレる可能性。README の表は自社推定。
  • 統合テストが 0 件: tests/ は fixtures のみ。#[test] 1669 件はすべて単体。「フィルタ通したら本当に削減されるか／重要情報を落としていないか」の回帰保証なし。
  • Claude hook が permissionDecision: "allow" を発行 (hooks/claude/rtk-rewrite.sh:92-100)。ユーザが "git *" のような広い allow を書いていると、エージェント確認なしに rtk 別バイナリが走る。信頼境界が静かに広がる。
  • openclaw に shell injection のグレーゾーン: openclaw/index.ts:29 の execSync(`rtk rewrite ${JSON.stringify(command)}`) は double-quote 内の $(...) / バッククォートが評価される。spawn/execFile でリスト渡しすべき。
  • unwrap/expect/panic! が src/ に 828 件。SECURITY.md:113 で自認している DoS リスクを守れていない。panic = "abort" (Cargo.toml:49) と相まって proxy 経由で本来動くべきコマンドが死ぬ可能性。
  • DISCLAIMER.md と実装が逆: DISCLAIMER は「デフォルト収集・無効化可」と書くが実装は「デフォルト OFF・opt-in」。コンプラ部門が混乱する類の不整合。

低

  • 9 種エージェント hook の保守負荷: codex / cline / windsurf は "rules file" 由来の prompt 注入のみで実質フィルタしていない (hooks/README.md:53-55)。

メタ情報の所感

  • 3.5ヶ月で 41,894 stars は organic ではあり得ない。ripgrep / fd / bat 等の同等規模 CLI は数年で到達する数字。多言語 README、update-readme-metrics.sh、Discord、homepage 等のマーケ装備からして Product Hunt 系・X キャンペーン・インセンティブ star・bot star のいずれかが濃厚。コード自体は 28k 行で実体はあるが、星数は信用しない方がよい。
  • openclaw/ は claw ≒ Claude を意識した命名。Anthropic から指摘される余地のあるトレードマーク的グレー。

採用判断

  • 試してもよい: 個人開発で、(1) Claude Code の広い auto-allow を書かない、(2) install.sh を使わず cargo install --git で自前ビルド、(3) telemetry を opt-in しない、(4) "削減量" 表示を額面通り信じない、(5) フィルタ後出力に重要情報が落ちていないか毎回確認、ができる人。
  • 触らない方がいい: 企業環境（ライセンス分裂で SBOM が割れる）、curl|sh を CI に入れる運用、エージェントの permission gate を信頼している現場、セキュリティ監査が入る開発。星数だけで採用すると「README の約束の半分しか実体がない」と気付くまで時間を溶かします。
