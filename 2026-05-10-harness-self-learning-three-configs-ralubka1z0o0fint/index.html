1. なぜ「ハーネスの自己学習」なのか

ハーネス工学（harness engineering）は Agent = Model + Harness という枠組みで、モデル本体ではなく「モデルを取り囲むツール・コンテキスト・フィードバックループ」を設計する営みを指す（Birgitta Böckeler / Martin Fowler, 2026-04）。

Böckeler は outer harness が果たすべき役割を 2 つに整理している:

  1. Guides（feedforward control） — 行動前に方向付ける（AGENTS.md, skills, code mods 等）
  2. Sensors（feedback control） — 行動後に観測して自己修正させる（lint メッセージ、構造テスト、AI レビュー等）

片方しかないと、フィードバックなしで同じミスを繰り返すか、ガイドを書いただけで効果検証ができないかのどちらかになる（Fowler 記事の主張）。

「自己学習」とは要するに Sensor で観測した失敗を蓄積し、次の Guide に転化する steering loop を回し続けること。Anthropic 自身も長期実行アプリ向けのハーネス設計記事で、generator と evaluator を分離し、評価結果をフィードバックすることがブレークスルーになったと報告している（Anthropic Engineering, 2026-03）。

ところが、3 段ロケットの自己改善基盤（30+ skills、40+ hooks スクリプト、claude-flow MCP の pattern-consolidator.sh 的な仕組み, dev.to 事例）はオーバーキルになりがち。最小構成で steering loop を閉じる には、実は 3 つの設定で足りる、というのが本レポートの主張。

3. 例1: Stop hook で「セッションの教訓」を memory に append する

何を解決するか

Sensor 側の最弱点は 「学んだことが揮発する」 点。会話を /clear した瞬間、せっかくユーザーから受けた訂正・ベストプラクティスは消える。Claude Code の auto memory 機能（~/.claude/projects/<project>/memory/MEMORY.md に index、配下の *.md に個別エントリ）を使えば、セッション横断で永続化できる。

ただし auto memory は「重要な情報があったら自動保存」がベース。Stop hook を使って「セッション終了時に必ず棚卸しする」契機を強制すると取りこぼしが激減する。

設定（~/.claude/settings.json）

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/end-of-session-memory.sh"
          }
        ]
      }
    ]
  }
}

スクリプト（~/.claude/hooks/end-of-session-memory.sh）

#!/usr/bin/env bash
# Stop イベントで stdout に出力したものは Claude にコンテキストとして渡される。
# ここで「棚卸し用プロンプト」を流し込み、memory への保存を促す。

cat <<'EOF'
[hook: end-of-session-memory]
このターンが本セッションの最終ターンなら、以下を点検してから終了せよ:

1. ユーザーから受けた訂正・好み・前提知識のうち、まだ memory に書いていないものはあるか？
   - あれば feedback / user / project / reference のいずれかとして memory ファイルを作成し
     MEMORY.md に 1 行 index を追加する。
2. このプロジェクト固有の事実（締切、関係者、外部システム参照先）で
   会話履歴にしか残っていないものはないか？
   - あれば project / reference 型として保存する。
3. 既存メモリと矛盾する情報を観測したか？
   - あれば該当ファイルを update / 削除する。

判断は厳しめに。「保存するほどでもない」と思ったらスキップして良いが、
「次回これを覚えていなければ同じ修正を受ける」と感じたものは必ず保存する。
EOF

なぜシンプルか

  • 既存機能を起動するだけ — auto memory システム自体は Claude Code に組み込み済み。書いていないのはトリガーだけ
  • shell 5 行 — メンテ対象が事実上ない
  • 失敗しても無害 — hook が落ちてもセッションは継続する

自己学習効果

毎セッション末に「今回のフィードバックを未来に残すか？」を自問させるので、1 回受けた指摘を 2 回受けない が原則として成立しやすくなる。Böckeler 流に言えば「Sensor で観測した失敗が蓄積層にコミットされる」段階の自動化。

4. 例2: UserPromptSubmit hook で「危険ワード検知 → 注意書き注入」

何を解決するか

蓄積された知識（CLAUDE.md, memory, rules）はセッション開始時にロードされるが、人間でもいつもの注意書きをスクロールせず読み飛ばす のと同じで、コンテキスト中盤以降は薄れがち。

特に「main ブランチへの force push」「rm -rf」「本番 DB の DELETE」のような 不可逆操作 は、ユーザー側のうっかりプロンプトをトリガーにして暴発する。プロンプト送信の瞬間に その文脈に最も関連するルールだけ を inject すれば、薄れを抑えられる。

これは Böckeler の Guide（feedforward control） に該当し、 dev.to 記事の SkillActivationHook パターン（キーワード→skill コンテキスト注入）を最小化したもの。

設定（~/.claude/settings.json）

{
  "hooks": {
    "UserPromptSubmit": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/inject-guards.sh"
          }
        ]
      }
    ]
  }
}

スクリプト（~/.claude/hooks/inject-guards.sh）

#!/usr/bin/env bash
# UserPromptSubmit hook の stdout は Claude のコンテキストに追加される。
# stdin からプロンプト本文を読み、危険ワードにマッチしたら該当ルールを inject する。

prompt="$(cat)"
rules_dir="$HOME/.claude/guards"

emit() {
  local rule_file="$1"
  [[ -f "$rules_dir/$rule_file" ]] || return
  echo "[guard: $rule_file]"
  cat "$rules_dir/$rule_file"
  echo
}

# 危険ワード → ルールファイルのマップ（最小構成）
echo "$prompt" | grep -qiE 'force[-_ ]?push|push --force|push -f'  && emit force-push.md
echo "$prompt" | grep -qiE 'rm[[:space:]]+-[rRf]+|rmdir[[:space:]]+/'  && emit destructive-rm.md
echo "$prompt" | grep -qiE 'drop[[:space:]]+(table|database)|truncate '  && emit destructive-sql.md
echo "$prompt" | grep -qiE 'reset --hard|checkout \. *$|clean -[fdx]+'   && emit destructive-git.md
echo "$prompt" | grep -qiE '本番|production|prod-|prod\b'                && emit production-touch.md

ルールファイル例（~/.claude/guards/force-push.md）

[force-push 検知]
- main / master への force push は事前に明示確認を取る
- 必要性を分解: rebase 結果の上書きなら branch.<name>.rebase 設定で代替できないか？
- どうしても必要な場合は --force-with-lease を提案する

なぜシンプルか

  • shell 一発 — grep で十分。LLM 呼び出しすら要らない
  • ルールが増えても 1 行追加で済む
  • ガード文字列は人間が増やしていけば良い — レビューで「またそれかい」と言われたパターンを追加していくだけ

自己学習効果

例 1 の memory 蓄積で「やりがちなミス」が観測されたら、対応するガードファイルを ~/.claude/guards/ に 1 枚足す。Sensor → Guide の橋渡し がここで起こる。dev.to 事例の 40+ スクリプトはこの発展形。

5. 例3: 実装と評価のエージェント分離（generator vs evaluator）

何を解決するか

Anthropic 自身がフロントエンドデザインの長期実行ハーネス開発で報告した最大の発見は、

「自己評価させると LLM は自分の作った成果物を例外なく褒める」

ということ。これは subjective なタスクだけでなく、テストが通る通らないが明確なタスクでも生じる。解決策は単純で、生成者と評価者を別エージェントに分離し、評価者にはあえて懐疑的な rubric を持たせる。

これは Claude Code では subagent（Agent ツール） で素直に実装できる。CLAUDE.md に「自分は実装、評価は別のエージェント」と書くだけ。本レポートの読者である横田氏の global CLAUDE.md には既にこの方針が書かれているが（「計画と評価は subagent に委譲、実装は自分」）、効果を最大化するには rubric を skill / 別ファイルに切り出して評価エージェント専用化する のが鍵。

設定（~/.claude/CLAUDE.md の該当節）

## 開発タスクのエージェント分担
- 計画フェーズ（要件定義・仕様設計）はサブエージェントに委譲する
- 評価フェーズ（QA・コードレビュー）はサブエージェントに委譲する
- 実装フェーズは自分で行う
- 理由: 自己評価は甘くなる傾向があるため、生成と評価を分離する

## コードレビュー時の心構え
- LLM は自分（または他LLM）のコードを甘く評価しがち。判断に迷ったら厳しい側に倒す
- バグを見つけたらどれほど小さく見えても報告する
- 「動いているからOK」ではなく「仕様通りに動いているか」で判断する
- 詳細な観点は `~/.claude/code-review-rubric.md` を参照

rubric の最小例（~/.claude/code-review-rubric.md）

# Code Review Rubric

評価エージェントは以下を厳しめに採点する。**「概ね良い」は禁句**。

## 機能性
- [ ] 仕様書（または会話で合意された要件）と実装の対応が 1:1 で確認できるか
- [ ] エッジケース（空入力 / null / 上限超過 / 並行アクセス）の扱いが明示されているか
- [ ] エラー時のユーザー観測動作が定義通りか

## 不変条件
- [ ] 関数の事前条件・事後条件・例外を破る入力で挙動が破綻していないか
- [ ] 副作用の発生順序が依存性に対して正しいか

## 退行
- [ ] 修正対象以外の挙動が変わっていないか
- [ ] 削除されたコードの呼び出し元が網羅的に追えているか

## 採点ルール
- 1 つでもチェックが付かなければ「修正必要」を返す
- 「実害は小さい」「後で直せる」は判断材料にしない（実装側の責務）

subagent への投げ方（運用パターン）

実装完了時に必ず:

Agent ツールで code-reviewer subagent を起動 →
prompt: "以下の差分を ~/.claude/code-review-rubric.md に従って厳しめに採点。
        実装意図は <こうこう>。" → 差分本文

なぜシンプルか

  • 追加インフラなし — Agent ツールと markdown 2 枚だけ
  • rubric が学習対象になる — レビューで見落とした観点を rubric に追記する steering loop を、人間が回す

自己学習効果

評価で挙がった指摘のうち「次回以降も再発しそうなもの」は、

  • ルールレベル: rubric の項目を追加
  • コードレベル: lint / 構造テストに昇格させる（Böckeler 言う Computational sensor 化）
  • プロンプトレベル: 例 2 のガードファイルを追加

の 3 経路で蓄積層に還流する。評価結果を「批評で終わらせない」のがポイント。

7. 段階的に導入する順序

優先順は 例1 → 例3 → 例2:

  1. まず例1: Sensor の蓄積基盤がないと、何を Guide に書くか決められない
  2. 次に例3: 評価が甘いままだと、例1 で蓄積する内容が「実は間違いだった memory」で汚染される
  3. 最後に例2: 例1, 例3 で「またこれか」のパターンが 3 つ以上見えてきてから初めて意味を持つ

dev.to 記事の 30 skills セットアップは「全部入りで始めるな、3 つから」と明言している。本レポートの 3 例は その「最初の 3 つ」のテンプレート として読める。

8. 落とし穴

  • 例1 の memory 肥大化: MEMORY.md は本レポートのコンテキスト規約上 200 行で truncate される。蓄積が進んだら定期的に統合 / 削除する pattern-consolidator 的な作業が必要（最初は手作業で OK）
  • 例2 の grep 偽陰性: 「rmdir」と「mkdir」のように似た文字列を弾きすぎると邪魔になる。通知だけに留め、ブロック（exit 1）は最後の手段に
  • 例3 の reviewer-LLM bias: 評価エージェントも結局 LLM。Anthropic の知見通り「skeptical 寄りに calibrate する」rubric が必要。同じモデル同士だと特に甘くなるので、評価役には別モデル / 別 system prompt を充てる
  • 3 つすべてに共通: ハーネスのコードは production コードと同じ規律で扱う。バグった hook はバグったセッションを量産する

10. 一言まとめ

自己学習ハーネスは「観測（Sensor）→ 蓄積 → 注入（Guide）」の 3 工程。それぞれに 1 つずつ最小設定を置けば、ハーネスは静かに賢くなり続ける。
