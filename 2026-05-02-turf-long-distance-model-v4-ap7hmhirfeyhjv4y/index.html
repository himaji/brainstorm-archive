1. 背景

  • v3_wide は LightGBM ではなく config.yaml のウェイト + Python の手書きスコアリングモデル（スピード指数 + 上がり3F + 騎手 + 血統 + ペース補正 + ワイド EV 計算）。
  • v3 は本番で運用されているがバックテスト未実施。ユーザーも ROI 1.0 超えに懐疑的。
  • Turso 本番 DB に重賞のオッズ・予想・結果が 2021–2025 で揃っており（578 レース）、バックテスト基盤の前提条件は整っている。
  • スクリプト scripts/ にバックテスト集計ツールは未実装。

2. スコープ

  • 対象: 芝 2400m 以上のレース（重賞のみ、Turso DB 既存データに準拠）
  • 非対象: ダート、芝 2400m 未満、新馬・未勝利・条件戦
  • 明日朝までに完了させる: タイムボックス。ML 化（LightGBM 等）はスコープ外（次回以降）
  • 投入判断: 自動デプロイ条件は「out-of-sample バックテスト ROI 1.0+」。未達の場合は天皇賞春は v3_wide のまま運用

3. 成功基準

項目 基準
必須: out-of-sample (2025) ROI 1.00 以上（未達なら投入しない）

参考: 推奨買い目数 / レース 0 件にならないこと（モデルが何も推奨しない退化を避ける）

ROI 計算は Odds.betType='wide' の確定オッズ × 推奨 bet_amount で行う。look-ahead を避けるため、推奨時点 odds (alternativePicks[*].odds) はバックテストでは使わない。

4. アプローチ

Phase 1: バックテスト基盤の構築（共通インフラ）

scripts/backtest-wide.ts を新規作成：

  • 入力: --model <model_id>, --surface 芝, --min-distance 2400, --from YYYY-MM-DD, --to YYYY-MM-DD
  • 処理: 該当レースの Prediction.alternativePicks を取得 → 各 pick (pick1.number, pick2.number, bet_amount) を抽出 → RaceResult で 1〜3 着判定 → Odds(betType=wide) で確定オッズ参照 → 投資額・払戻額・的中数を集計
  • 出力: 全体 ROI / 的中率 / 推奨数、年別・距離別・オッズ帯別の層別表
  • v3 にも v4 にも使えるように汎化する（model_id をパラメータ化）

Phase 2: v3 ベースライン測定 + 失敗パターン抽出

  • v3_wide で 2021–2024 (Train) と 2025 (Test) の両方を測定
  • 抽出すべき失敗パターン例：
  • 距離帯別 ROI（2400 / 2500 / 3000+）
  • 競馬場別 ROI（京都 / 阪神 / 東京 / 中山 …）
  • オッズ帯別 ROI（〜10 倍 / 10〜30 / 30+）
  • 馬場状態別 ROI（良 / 稍重 / 重 / 不良）
  • 「v3 が高 confidence で外したレース」と「拾えなかった的中レース」のリスト
  • 出力: docs/research/v3_wide_long_distance_baseline_2026-05-02.md

Phase 3: v4_long_wide 実装

ディレクトリ構成：

  • A 戦略: v3 のロジックをコピーして config.yaml のウェイトを芝 2400m+ 向けに最適化
  • 最適化対象: factor_weights.*, parameters.temperature, parameters.ev_threshold
  • 手法: grid search（離散的なウェイト候補）で Train set の ROI を最大化
  • 最適化スクリプト: scripts/tune-v4-long.ts（または Python）
  • B 戦略（A で ROI 1.0+ 未達なら追加）: ロジック修正
  • 候補1: 距離信頼度 _distance_trust の傾きを長距離向けに緩く（2400m+ でも過去 1600m 走を使えるように）
  • 候補2: cornerPositions（4 角通過順）を position_score に追加（最近スクレイプ済み）
  • 候補3: スタミナ指標 = 過去 2400m+ 走の上がり 3F の安定性
  • Phase 2 で出た失敗パターンを見て候補から優先度付け

Phase 4: v4 バックテスト & 投入判断

  • v4 で予想生成 → Phase 1 のバックテストで Test set (2025) の ROI を測定
  • ROI 1.0+ → scripts/run-predict.ts --model v4_long_wide で天皇賞春に予想生成 → DB 反映 → main merge → CF Pages 自動デプロイ
  • ROI < 1.0 → v3_wide のまま運用、knowledge は memory & 設計書に残して次戦に向けて再挑戦

6. 採点（評価）の正しさ

  • ROI 計算は 投資額の合計（bet_amount の総和） を分母に、的中時の払戻額（bet_amount × 確定 wide オッズ） を分子に
  • bet_amount は Prediction.alternativePicks[*].bet_amount を信頼する（kelly 計算済の値）
  • ワイドは 1 着 + 2 着 / 1 着 + 3 着 / 2 着 + 3 着 のいずれかにヒットすれば的中。3 通りそれぞれの確定オッズが Odds テーブルにある
  • 同着・除外の取扱いは v1 では考慮しない（極小ケース、影響を切り捨て）

7. リスクと却下案

リスク 緩和策
Phase 1 のバックテストに想定外バグ → 全体的に数値が信用できない サンプル 1 レースで手計算と突き合わせる単体検証ステップを Phase 1 に組み込む
Phase 2 の失敗パターン分析でサンプル不足（2400m+ 重賞は年 25 程度） 距離帯を細かく切りすぎず、「2400 / 2500–2999 / 3000+」の 3 区分に止める
v4 が Train set にオーバーフィット（grid search の典型病） Test set の ROI で投入判断する。Train ROI が高くても Test < 1.0 なら不採用
時間切れで明日朝までに終わらない Phase 1 → 2 まで完了して v3 のベースラインだけは握る。v4 投入は諦め、知見は次戦へ

却下案: - ML レイヤー追加（meta model）/ 完全 ML 化: 1 日では検証品質が担保できない。次回以降 - v3 の本体改修（v3 を直接いじる）: 既存運用へ影響。新モデル v4 として並列構築し、置き換えは投入判断後
