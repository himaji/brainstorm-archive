追記 — topK=5 を外したらどうなるか

本番 DB に保存された S2 picks は 5 点。これは predict.py 内の S2_TOPK = 5 による cap。pairs_top_ev (top 30 by EV) と DB 内の全 105 ワイドオッズから、jointP × R で計算された jointP / EV を全 nC2 ペアで再現すると以下になる:

22 点の中に 的中ペアが 2 件 含まれていた:

  • 11 位 3-15 (jointP=0.0552, ev=1.97, ワイド 32.3-39.0 倍) ★HIT
  • 21 位 7-15 (jointP=0.0510, ev=1.83, ワイド 32.5-39.3 倍) ★HIT

100 円ずつなら投資 2,200 円 / 払戻 約 6,500-7,200 円 で ROI 約 295%。1-3 着の 3 頭中 2 頭を絡めた配当 30 倍超のワイドを 22 点に圧縮できていたが、topK=5 で先頭 5 件 (全て #4 軸) のみ採用された結果、的中ペアは取りこぼされた。

topK=5 はどこから来たか

履歴を辿ると PR #109 (C-2 bet selection sweep) の最良 cell をそのまま採用したもの。当時の sweep grid:

topK = {1, 2, 3, 5}
evMin = {0.0}
jpMin = {0.0, 0.03, 0.05, 0.08}
maxHorseOdds (mho) = {30, 50, 100, 1000}

11,980 cell 走らせた中で S2 ベスト cell = topK=5 / jpMin=0.05 / mho=30 / msp=10 (test_n=56 / hit=7 / test_roi=2.16 / CI 下限 0.76)。これを predict.py に直接ハードコードした。

その後の修正で topK=5 の意味が変質

採用後、運用上の問題で 3 つの constraint が外/緩されている:

PR変更理由
#112maxHorseOdds 制約 30 → ∞ (撤廃)人気集中レースで picks=0 を回避
#114ev ≥ 1.0 floor 追加期待値マイナス pick 排除
#115jpMin 0.08 → 0.05pastResults 補完で jointP 分布が下方シフト

結果として現状の S2 は 「sweep で最良 cell に到達した時の constraint セット」とは別物 になっている:

  • 当時の cell では mho≤30 / msp≥10 の縛りで 候補ペア数自体が小さかった (大半のレースで qualify するペア ≤ 5) → topK=5 cap は binding していない だけで、cell の真の効果は jpMin/mho/msp の AND だった可能性が高い
  • PR #112 で mho 制約を外したことで 候補ペアが爆発的に増えた (今回 22 件) → 突然 topK=5 cap が binding に変質し、jointP 同率付近で恣意的に切り捨てる装置 になった
  • 今回の天皇賞春は jointP=0.0552 が 12 件 tied で、その上位の 0.0667 (3 件) と 0.0722 (2 件) を含めても上位 17 件まで「ほぼ同率」。ここから先頭 5 件を取るのは精度ではなくほぼランダム抽選

含意

  • S2 の sweep 最良 cell は再検証が必要: PR #112/#114/#115 の修正後に topK の感度を sweep し直した記録がない。「mho=∞, msp=0, evMin=1.0, jpMin=0.05」で固定し topK ∈ {5, 8, 10, 15, 20, ∞} を sweep して CI 下限の topK 依存性を確認すべき
  • 単発レース 1 件で topK 拡張は決められない: 「天皇賞春で 5→22 にすれば的中したから topK 上げよう」は典型的な selection bias。期待されるべきは「topK を緩めた cell が test 期間全体で 同等以上の CI 下限を出すか」
  • 低コストの並走運用が妥当: 当面の picks は現状の topK=5 のまま、forward test 期間 (6 ヶ月想定) は 「topK=∞ かつ qualify 全部買い」シミュレーション値を別途記録。両者の forward ROI を比較してから sweep 再走と picks 仕様変更を一括で決める
  • jpMin=0.05 の足切りが緩いのも合わせて再検討: 22 件中 11 件は jp=0.0552 / 0.0510 で閾値ギリギリ。jpMin を 0.06 / 0.07 に上げると tied 群を一段カットできるが、これも sweep で確認すべき

主要な発見

  • 本番現行 (topK=5) は test 期間で実際に損失: 87 bets / 6 hits / ROI 0.952 で -420 円。PR #109 の sweep 当時 (mho=30 / msp=10 込みで test_n=56 / ROI 2.16 / +6,510 円) と同じ topK でも別物になった。
  • topK=3 が現行 constraint 下では最良: ROI 1.267 / CI 下限 0.296 / +1,440 円。topK=5 より「絞った方が儲かる」 → jointP 順位の信頼性は上位 3 件までしか効いていない。
  • topK を増やすほど崩壊: 8 → ∞ で hits 数は 6→8 に微増するが bets が 129→212 に膨らみ、ROI と P/L は単調悪化。「topK 外せば的中ペアが増える」は今回のレース固有の selection bias で、test 期間全体では成立しない。
  • 天皇賞春の「topK=22 で +4,300 円」は 1 レース逸話: 同じ条件を test 期間に適用すると ROI 0.607 / -8,330 円。先の追記で警告した「単発レースで topK 拡張は決められない」がそのまま実証された。

含意 — 構造的な問題

  • PR #112 / #114 / #115 の累積で本番が劣化している可能性: 個別 PR は妥当な修正に見えるが、test 期間ベースでは重ね合わせで ROI 2.16 → 0.95 に落ちている。「sweep で見つけた最良 cell の周辺は最適化されているが、cell 外に出た瞬間に landscape が変わる」典型。
  • 本来取るべきだった対応: PR #112 (mho 撤廃) の時点で、constraint を変えた以上 topK / jpMin も同時 sweep し直すべきだった。1 軸ずつ patch した結果、cell 全体が最良点から離れた。
  • 当面の暫定運用案 (どれも単独では決められない、要ユーザー判断):
    • (a) topK を 5 → 3 に戻す (test ROI 0.95 → 1.27、ただし test_n=54 で CI 下限 0.30 と弱い)
    • (b) jpMin を 0.05 → 0.08 に戻す (PR #115 前に巻き戻し、test ROI 0.95 → 1.27 / CI 下限 0.34)
    • (c) jpMin=0.04 + topK=∞ に切り替え (test ROI 1.30 / CI 下限 0.47 / 最大の P/L だが picks 数 16 倍化、運用負荷)
    • (d) C-2 sweep をフルに再走 (mho/msp/topK/jpMin 同時 grid) し、現行 landscape での最良 cell を取り直す ← 本筋
  • 天皇賞春の「topK 外せば当たった」は罠: 単発の的中の裏に、test 期間全体での損失拡大が隠れていた。当初の追記で「単発レースで決められない」と書いた懸念が、まさにこの sweep で具体的な数値として裏付けられた。

結論として、天皇賞春の picks 取りこぼしは「topK の問題」ではなく「PR #112/#114/#115 の累積で cell が最良点から離れたこと」の症状。topK を緩めても解決せず、constraint 全体を sweep し直す必要がある。
